# 知识库 Agent 使用导览

本文定义 `data/knowledge/` 的查询契约：调用方提供定位输入或检索目标，知识库返回可引用资料、上下文、关系链、别名导航、消歧线索和证据路径。具体输出由调用方处理。

## 输入边界

用户不会提供 `choice key`、`video key`、`profile key`、`tag key` 或 `item id`。用户通常只会表达自然语言问题，例如“这里发生了什么”“这个人是谁”“我现在这个状态该怎么理解”。

当查询需要定位当前游戏状态、当前剧情片段或当前选择页时，Agent 必须从后台可读来源或外层输入取得定位信息：

- 真实游戏日志或运行探针。
- 存档、模拟器 state、HTML 本地状态或外层运行器传入的结构化状态。
- 已解析的路线图、节点图、当前路线记忆和选择记忆。
- 批处理、测试或构建流程传入的目标 key。

如果定位信息读取不到，正确行为是报告“缺少定位输入”或“缺少运行状态输入”，而不是要求用户提供 key，也不是凭记忆猜测。

本知识库不假设 Agent 能直接看用户屏幕。只有外层运行器显式传入 OCR、截图识别、日志解析或游戏进程结构化数据时，这些内容才可以作为输入，并且必须标注为外层采集结果。

## 目录职责

- `video_subtitles/docs/<videoKey>.subtitles.md`：按 video key 组装后的剧情片段文档。包含视频 key、章节/线路、前置选择、前后节点、时间码、说话人和字幕正文。需要理解具体视频内容时，这层是目标片段事实主来源。
- `video_subtitles/INDEX_BY_CHAPTER.md` 和 `video_subtitles/INDEX_BY_ROUTE.md`：字幕片段的章节索引和线路索引。需要按章节或线路进入时先读对应索引，再打开具体字幕文档。
- `storyline_guide.md`：路线图数据使用指南。说明 `storyline_graph_data.json`、`nodes.jsonl`、`edges.csv` 的职责、字段和常见查询方式。
- `task_guides/`：具体任务的调用导引。导引只说明取证方法和输出注意事项，不作为剧情事实来源。
- `dossiers/characters/*.md`：人物档案。包含身份、路线语境、关系链、别名、检索锚点和可引用正文。
- `dossiers/items/*.md`：物品、概念、制度和典故档案。包含概念解释、关系链、检索锚点和可引用正文。
- `dossiers/aliases/`：固定别名、称号、旧 key、阶段导航和上下文称谓消歧指南。先读该目录下的 `README.md`、分类 README 或具体 `.md` 文件；不要把目录路径当作具体文档读取请求。命中固定别名时继续读取它指向的人物档案；命中上下文称谓时先读消歧指南。
- `dossiers/unlocks/*.md`：档案解锁线索，用于理解资料出现时机。解锁线索不能替代档案正文，也不能单独证明当前剧情事实。
- `dossiers/reference/`：前作/旧作、未实装候选或旧名溯源参考资料。只在查询目标需要这些背景时读取，不能覆盖当前游戏档案正文。
- `dossiers/unresolved/`：只有锚点、没有正文的补采队列。默认不作为事实依据，只用于补采、映射排查和索引占位。
- `video_summaries/`：AI 生成的派生摘要层。可用于补充上下文、理解长距离前因后果、回顾前文和辅助判断路线位置；需要核验目标剧情事实时，仍应回到字幕片段、路线图和官方字段。

## 查询取证优先级

处理具体问题时，先确定查询目标，再按证据强度取资料：

1. 定位输入：后台日志、存档、HTML state、外层运行器状态、批处理目标或测试目标。
2. 目标视频字幕片段：`video_subtitles/docs/<videoKey>.subtitles.md`。
3. Storyline 路径图：先读 `storyline_guide.md` 确认文件职责，再查询节点顺序、前置选择、兄弟选项、直接前后视频、合流和分支关系。
4. 同路线前文摘要：用于补充前因后果，不覆盖目标片段字幕、路线图或档案正文。
5. 人物和词条档案：用于身份、关系、动机、物品和制度解释。
6. 官方摘要字段：作为开发侧参考；若和目标字幕冲突，以目标字幕和路线图为准。
7. 前作/旧作参考文本：只解释过往设定、旧名或未实装候选，不覆盖当前游戏文本。
8. `unresolved/`：仅用于补采或排查，不作为事实依据。

## 检索锚点怎样使用

档案中的 `profile key`、`tag key`、`item id`、TID 和 video key 是机器检索锚点，不是写给用户看的剧情内容。

使用方式：

- 从运行状态、任务目标或路线图读到 node/video/choice 信息后，用 key 精确打开对应字幕片段或路线节点。
- 从字幕、路线节点、官方字段或结构化状态中识别人物名、物品名、tag/profile 字段后，检索 `dossiers/characters/` 或 `dossiers/items/`。
- 如果识别到的是固定称号、身份名或旧 key，先查 `dossiers/aliases/README.md`、`dossiers/aliases/characters/README.md` 或具体别名 `.md`，再读取导航指向的人物档案。别名文件只负责跳转，不是人物正文来源；不要把目录路径当作具体文档读取请求。
- 同名人物、旧名、规避名和路线身份必须通过 key、路线、章节和正文语境共同消歧。
- `皇后`、`陛下`、`殿下`、`楼主`、`姐姐`、`主子` 等关系/职位/阶段称谓不是全局固定别名；先读 `dossiers/aliases/characters/contextual-title-guide.md`。其中“说话人 + 称谓”可作为强线索，例如袭心/小佑子/蕊儿说“主子”通常指伍媚娘，但仍要结合当前字幕、路线和前后节点确认。
- 如果只命中文件名但正文没有相关内容，不要把文件名当作证据。
- 命中人物或词条档案后，应读取完整 Markdown 文档；不要用关键词脚本摘句、裁剪正文或替模型判断哪些段落“相关”。
- 工具可以负责定位、召回、导航和校验证据路径；剧情含义、人物关系、动机和风险解释必须由模型基于完整资料、目标字幕和路线图判断。

## 派生摘要边界

生成或重写某个目标视频摘要时，可以使用已生成的同路线前文摘要作为辅助上下文；摘要层必须被视为 AI 生成的派生材料，不能替代目标视频字幕、路线图和档案正文。

使用摘要层时先做过滤：

- 排除当前目标视频自己的既有摘要；不能读取、复述或改写目标自己的摘要来完成目标摘要。
- 同路线前文摘要可以作为辅助背景。
- 排除未完成、已知错误或来源不明的摘要。
- 同路线前文摘要可作为背景，尤其适合补充长距离前因后果。
- 后文摘要和兄弟分支摘要只能用于理解路线位置、后续去向和分支差异；不得把后文细节写进当前视频摘要，也不得制造剧透式事实外泄。
- 如果摘要和字幕、路线图或档案正文冲突，以字幕、路线图和档案正文为准，并把冲突写入风险提示。

无法确认摘要来源、路线位置或证据边界时，直接回到 `video_subtitles/docs/*.subtitles.md`、路线图和档案正文。

## 路线人物档案怎样用

人物档案里的路线小节表示这段档案文本的解锁来源和剧情语境，不表示人物只存在于该路线。

使用规则：

- 如果当前路线有对应小节，优先读取当前路线小节。
- 如果支线没有专属小节，可以读取主线里的基础身份、职务、亲属关系和初始阵营来辅助消歧。
- 不要把主线后续事件、心态变化、结局、死亡或关系变化自动当成支线已发生事实。
- 如果人物在主线和支线都有小节，通常说明该人物在两条线里的经历、身份、关系或结局存在差异，必须按当前路线分开引用。
- 可以引用不冲突的基础身份信息，但必须让目标字幕、路线图和当前运行状态优先。

## 查询流程

1. 明确查询目标：目标可以是当前运行状态、某个 video key、某个 choice key、某个人物/物品、某段字幕、某个剧情问题或外层传入的结构化对象。
2. 打开最直接的事实文档：具体视频先读 `video_subtitles/docs/<videoKey>.subtitles.md`；具体人物/物品先读对应档案；路线问题先读路径图和相关字幕片段。
3. 扩展相邻上下文：根据 `storyline_guide.md`、路线图和字幕索引读取前置选择、兄弟选项、直接前后节点、合流关系和同路线前文。
4. 扩展背景知识：读取涉及人物、物品、制度、别名、解锁线索和必要的前作参考。
5. 保持证据边界：区分当前已发生事实、同路线背景、兄弟分支、坏结局、前作参考、候选资料和未恢复条目。
6. 返回证据包：把可引用资料、上下文、关系链和证据路径交给调用方；输出形态不在知识库内预设。

## 接入约定

- 目标 key、当前状态或待处理片段来自外层输入，不来自用户手工提供。
- 需要当前游戏状态时，外层运行器应传入当前章节、线路、节点或 video key、当前选择问题、选项文本、已选项和可选项、路线记忆、数值和条件判断结果。
- 需要处理固定视频或资料时，批处理器应传入目标 key，并让 Agent 按查询流程扩展上下文。
- 任何调用方都不能把兄弟分支、坏结局、历史探索路线或前作参考混成当前路线事实。
- 输出结论时应能回指到字幕片段、前置选择、路线节点、档案正文、官方摘要或可用摘要。

## 档案状态说明

人物和物品档案按事实可用性分成三类：

- `dossiers/characters/` 和 `dossiers/items/`：已经有可引用正文，可用于背景检索、人物/物品消歧和关系扩展。
- `dossiers/reference/`：前作/旧作、旧名溯源、未实装候选或外部背景参考。只有查询目标需要这些背景时才按需读取。
- `dossiers/unresolved/`：只有定位锚点，没有可引用正文，只用于补采、映射排查和索引占位。

人物档案里的正文小节是内容边界：

- `当前游戏档案文本`：当前游戏版本中的人物设定，优先用于解释身份、关系、阵营和动机。
- `当前游戏档案文本 - 女帝线/主线`：该段文本来自女帝线/主线剧情语境，可作为基础身份参考，但主线后续事件不能自动继承到支线。
- `当前游戏档案文本 - 新世界线/支线`：该段文本来自新世界线/支线剧情语境，当前处于支线时优先使用。
- `前作/旧作参考文本`：前作遗留或旧版本资料，只能解释过往设定，不能覆盖当前游戏文本。

## 禁止事项

- 不要要求用户提供任何 key。
- 不要要求用户截图来完成常规检索。
- 不要把 `unresolved/` 当成可引用资料。
- 不要默认读取 `dossiers/reference/` 作为当前游戏事实。
- 不要把前作/旧作参考文本覆盖当前游戏文本。
- 不要把兄弟分支、坏结局或历史探索路线混成当前路线事实。
- 不要把文件名、key、TID 写成剧情结论。
- 不要用原始 SRT 替代 `video_subtitles/docs/*.subtitles.md`。
- 不要让关键词、正则或候选表替代模型阅读全文后的判断。
- 生成目标视频摘要时，不要读取或引用目标视频自己的既有摘要；不要把错误或来源不明的摘要当成背景；不要把 AI 摘要当成一手事实来源。
